## 1. Overview & Introduction

The GDSC Farmingdale Links application acts as the official frontend for the Google Developer Student Clubs (GDSC) chapter at Farmingdale State College. It's designed to provide students, faculty, and interested community members with effortless access to upcoming events, past activities, and relevant resources.

### Purpose & Goals

Our primary goals with this application are:
*   **Centralization:** Create a single, easy-to-access platform for all GDSC Farmingdale-related information.
*   **Engagement:** Foster greater participation in events and community activities by making information readily available and engaging.
*   **Accessibility:** Ensure the application is responsive and usable across various devices and platforms.
*   **Modern Experience:** Deliver a smooth, interactive, and visually appealing user experience using cutting-edge web technologies.

### Problem Solved

In a vibrant academic community, information about events, workshops, and opportunities can often be scattered across multiple platforms. This leads to missed opportunities and reduced engagement. GDSC Farmingdale Links solves this by:
*   Aggregating event data from dedicated, reliable APIs.
*   Providing a consistent, user-friendly interface for browsing.
*   Enabling quick access to event details, resources, and sign-up links.

### Target Audience

This application is built for:
*   **Current GDSC Farmingdale Members:** To easily track upcoming meetings, workshops, and social events.
*   **Prospective Members:** To discover GDSC activities and learn how to get involved.
*   **Farmingdale State College Students & Faculty:** To stay informed about technology-related events and opportunities on campus.
*   **Community Members:** Anyone interested in GDSC initiatives and tech-related learning.

*   **Core Components (C):** Reusable UI elements (`ContentCard`, `ImageCarousel`, `videoCarousel`) and logical units (`AuthContainer`, `SearchBar`, `ThemeProvider`) that form the building blocks of the application.
*   **State Management (E):** Uses context API or libraries like Zustand (`src/core/store.ts`) for managing global application state, ensuring data consistency across components.
*   **Firebase Auth Module (F):** Handles user authentication logic, interacting with the Firebase Authentication Service.
*   **Utility Hooks (H):** Custom React hooks (`src/hooks/`) for common functionalities like managing clipboard actions, checking online status, or handling click-outside events.
*   **Firebase Authentication Service (I):** An external service by Google that provides secure user authentication (anonymous, Google Sign-in).
*   **Links API (SSR Web Service) (J):** An external, server-side rendered (SSR) web service responsible for fetching and providing event data to this frontend application. ([Details](https://github.com/WomB0ComB0/gdsc-farmingdale-links-api))

### Running the Application

#### Development Mode

To run the application in development mode with hot-reloading:

```bash
npm run dev
# or using Yarn:
# yarn dev
```

The application will typically be accessible at `http://localhost:5173` in your web browser. Vite will provide the exact URL in your terminal.

#### Production Build (for serving statically)

To create a production-ready build:

```bash
npm run build
```

This will compile the application into the `dist/` directory. You can then serve these static files using a simple HTTP server (e.g., `serve` npm package) or deploy them to a hosting service like Firebase Hosting.

```bash

# Serve the production build

```

For a containerized development experience or local production simulation:

1.  **Build the Docker image:**
    ```bash

    ```

2.  **Run the container:**
    ```bash
    docker compose up
    ```
    The application will be available at `http://localhost:5173`. Changes to your code will be reflected due to volume mounting.

## 6. Limitations, Known Issues & Future Roadmap

Every project has its constraints and areas for growth. Here's what to expect.

### Current Limitations

*   **API Dependency:** The application heavily relies on the availability and correct functioning of the external Links API and Links Links API.
*   **Client-Side Only Features:** As a frontend application, advanced server-side processing or direct database interactions are handled by the backend APIs, not within this repository.
*   **Limited Admin Functionality:** Currently, there is no integrated admin panel for content management directly within this application.
*   **Static Content (if APIs are down):** While robust, if backend APIs are entirely unreachable, dynamic content display will be affected. Fallback mechanisms are in place but may show limited data.

### Known Issues

*   **Offline Mode (Partial):** While basic offline caching for assets is supported via PWA, dynamic content requires an active internet connection to fetch the latest data.
*   **Image/Video Loading Errors:** Occasionally, external media links might fail to load due to source issues or network problems. (Handled by `Fallback.tsx` in `src/components/dom/`).

### Future Roadmap

We are continuously working to improve GDSC Farmingdale Links. Here are some planned enhancements:

*   **Enhanced PWA Capabilities:** Further development of Progressive Web App features for a more native-like experience (e.g., better offline support, notifications).
*   **User Profiles & Personalization:** Allow users to save favorite events, set reminders, or personalize their feed.
*   **Advanced Search & Filtering:** Implement more granular search options and intelligent filters for events.
*   **Integrated Feedback System:** Allow users to provide feedback directly through the application.
*   **Event Submission Form (Moderated):** Enable GDSC members to submit event proposals directly via the frontend, subject to moderation.
*   **Multilingual Support:** Support for different languages to cater to a diverse user base.

## 7. Contributing & Development Guidelines

    *   Delete `node_modules` and `package-lock.json` then `npm install` again.
*   **Environment variables not loading in production:**
    *   For production builds, environment variables prefixed with `VITE_` are typically inlined during the build process. Ensure your `.env` file is present *before* running `npm run build`.
    *   If deploying to Firebase, ensure environment variables are correctly configured in Firebase project settings or build scripts.
*   **Docker build/run issues:**
    *   Verify Docker Desktop is running.
    *   Ensure you are in the root directory where `Dockerfile` and `compose.yaml` are located.
    *   Check Docker logs for specific error messages: `docker compose logs`.
